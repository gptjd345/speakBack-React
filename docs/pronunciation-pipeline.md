# 발음 평가 파이프라인 설계

---

## 1. 설계 목표

본 시스템의 목표는 원어민처럼 발음하는 것(native-like)이 아니라,
원어민이 무리 없이 이해할 수 있는 발음(communicable pronunciation)을 만드는 것입니다.

> "Would a native speaker understand this without effort?"

이 목표에 따라 발음 평가의 우선순위를 다음과 같이 설정했습니다.

음소 정확도(segmental)보다 **강세·리듬·억양을 중심으로 평가**

이는 언어학 연구에서도, 강세와 리듬이 intelligibility에 더 큰 영향을 미친다는 점이 확인된 바 있습니다.

---

## 2. STT 기반 평가의 한계

위 목표를 달성하기 위해 기존의 STT 기반 평가 방식을 검토하였으나,
다음과 같은 구조적인 한계를 확인했습니다.

STT는 음성을 텍스트로 변환하는 과정에서 **음향 정보(prosody)를 모두 제거**합니다.

| 평가 항목 | STT로 가능 여부 |
|---|---|
| 단어 전달력 | △ (간접적으로 가능) |
| 강세 | ❌ |
| 리듬 | ❌ |
| 연음 / 축약 | ❌ |
| 억양 | ❌ |

이로 인해 다음과 같은 문제가 발생합니다.

- 또박또박 읽으면 높은 점수가 나오는 구조
- 자연스러운 발화(연음, 축약)가 오히려 불리하게 평가됨
- 실제 communicability를 반영하지 못함

따라서 STT만으로는 본 시스템의 목표를 만족시킬 수 없다고 판단했습니다.

---

## 3. 설계 결정: 음향 기반 분석 도입

위 한계를 해결하기 위해,
단순 텍스트 비교가 아닌 음향 기반 분석 파이프라인을 설계하였습니다.

핵심 전략은 다음과 같습니다.

- **Whisper** → 단어 단위 시간 정보 확보
- **librosa** → 해당 구간의 음향 특성 분석
- **GPT** → 최종 전달력 평가

---

## 4. 전체 파이프라인 구조

| 평가 요소 |
|---|
| 내용어 강세 |
| 평탄한 발화 감지 |
| 연음 |
| 발음 인식 오류 탐지 |
| GPT 기반 종합 평가 |

---

## 5. 음향 분석 설계

### 5.1 단어를 기준으로 발화 구간 분리

Whisper의 단어별 타임스탬프(word-level timestamp)를 활용하여
각 단어의 발화 구간을 정확히 분리합니다.

```python
timestamp_granularities=["word"]
```

### 5.2 에너지 기반 강세 분석

```python
rms = librosa.feature.rms(y=audio_segment)[0]
word_energy = np.mean(rms)
```

- 각 단어의 RMS energy 계산
- 문장 내에서 **상대 순위(rank)** 로 변환

**설계 의도**

| | 이유 |
|---|---|
| 절댓값 기준 ❌ | 화자마다 음량 차이 존재 |
| 상대값 기준 ⭕ | 문장 내 대비가 중요 |

---

## 6. 강세 평가 기준 설계

### 6.1 핵심 원칙

> "내용어가 기능어보다 충분히 강하게 발음되는가"

### 6.2 세부 기준

**1) 내용어 vs 기능어**

- 내용어의 에너지가 기능어 RMS 평균보다 낮은 경우 → 감점
- Python에서 이미 계산된 결과를 받아 판단 (일관성 향상)

**2) 평탄한 발화 감지**

```python
energy_cv = np.std(energies) / np.mean(energies)
```

- CV < 0.15 → 평탄한 발화로 판단
- 감점 및 리듬 피드백 제공

**3) 복합명사 강세**

```
SHOW dog  (O)
show DOG  (X)
```

→ 복합명사 내 1차 강세 위치 확인

### 6.3 평가 정책 요약

| 조건 | 처리 |
|---|---|
| 내용어가 기능어 평균보다 약함 | 감점 |
| 전체 에너지 평탄 | 감점 |
| 복합명사 강세 오류 | 감점 |
| 기능어가 약함 | 정상 |
| 연음 발생 | 긍정 피드백 |

---

## 7. 연음 분석

```python
gap = next_word.start - current_word.end
gap < 0.05  # 50ms 이하 → 연음으로 판단
```

**설계 의도**

- 자연스러운 발화 특징이므로 감점하지 않음
- 긍정 피드백 요소로만 활용

---

## 8. 발음 인식 오류 탐지

- Whisper 결과와 목표 문장 비교
- 불일치 단어 추출

```
thought → dought
```

**역할**

- 심각한 발음 오류 탐지
- 보조 지표로만 활용 (주 평가 아님)

---

## 9. GPT 기반 최종 평가

**입력 데이터**

- 단어별 에너지 순위
- 평탄한 발화 여부
- 연음 정보
- STT 불일치 단어

**출력**

- 점수 (0–100)
- 강점
- 개선점
- 리듬 피드백

**점수 기준**

| 점수 | 의미 |
|---|---|
| 90–100 | 명확하게 이해 가능 |
| 85–90 | 약간의 노력으로 이해 가능 |
| 70–85 | 이해 가능하나 불편 |
| < 70 | 이해 어려움 |

> 85점 이상 = communicable

## 10. 성능 문제 해결 (Numba JIT)

**문제**

- librosa pyin 최초 호출 시 5~15초 지연

**해결**

```python
@app.on_event("startup")
def startup():
    warmup_librosa()
```

- 서버 시작 시 미리 컴파일 수행
- 사용자 요청 지연 제거

---

## 11. 성능 최적화 (2단계 UI)

**병목 구간**

| 단계 | 시간 |
|---|---|
| STT / TTS | ~1초 |
| librosa 분석 | ~2초 |
| GPT 평가 | ~5초 |

**해결 전략**

- 목표 문장 확정 시 선처리
- `/prepare` API로 Redis 캐싱

→ 실제 분석 시 캐시 히트 → 응답 시간 단축

## 12. 결론

본 시스템은 단순 STT 기반 평가의 한계를 인지하고,
초기 설계 단계부터 communicable pronunciation을 목표로 설정하여

- 음향 기반 분석 (librosa)
- 강세·리듬 중심 평가
- GPT 기반 종합 판단

을 결합한 발음 평가 파이프라인을 설계하였습니다.

이를 통해 단순 정확도가 아닌,
실제 의사소통 가능성을 반영한 평가 시스템을 구현하였습니다.
